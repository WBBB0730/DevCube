## Problem Statement

我在 DevCube 里登记的项目如果一开始不是 git 仓库，之后在外面（或 Terminal 里）跑了 `git init`，Git Tab 仍然永远停在「该项目不是 Git 仓库」，只能移除项目再重新添加才能恢复；反过来 `.git` 被删除时，Git Tab 会卡在「无法加载提交」的错误态，重试也永远失败。而且 DevCube 里根本没有初始化仓库的入口——新项目想用 git 必须切出去敲命令；刚初始化的空仓库（0 个提交）在 Git Tab 里也只有一行提示，没法做首次提交。添加项目也只能选择已有文件夹，想从零开一个新项目得先去 Finder 建目录。

## Solution

三块能力打包：

1. **仓库状态自动跟进**：项目「是不是 git 仓库、仓库根在哪」不再是一次性判定——`.git` 出现或消失后，Git Tab 自动在「非仓库 / 空仓库 / 正常图谱」三态间切换，无需重新添加项目。
2. **初始化 Git 仓库**：非仓库兜底态提供「初始化 Git 仓库」按钮 +「刷新」按钮；初始化对话框可填初始分支名与远程仓库地址（均可留空）。空仓库有改动时图谱合成未提交更改行，提交面板全功能可用，首次提交后可顺势推送。
3. **添加项目双入口**：项目树的 "+" 变为菜单——「新建项目…」（系统面板选位置起名，创建新文件夹并登记）与「添加现有项目…」（原有流程）；拖入不变。

## User Stories

1. 作为开发者，我想在非仓库项目里跑过 `git init` 后，Git Tab 自动变成仓库视图，这样不用移除再重新添加项目。
2. 作为开发者，我想 `.git` 被删除后 Git Tab 回到「不是 Git 仓库」的兜底态，这样不会卡在无法恢复的报错里。
3. 作为开发者，我想在 Git Tab 的非仓库兜底态直接点「初始化 Git 仓库」，这样不用切终端敲命令。
4. 作为开发者，我想在非仓库兜底态有「刷新」按钮手动重新探测，这样自动检测万一失灵还有兜底。
5. 作为开发者，我想切回 Git Tab 时它顺手重新探测一次仓库状态，这样外部变化尽快反映。
6. 作为开发者，我想初始化时可以填初始分支名，这样一开始就用我要的分支名。
7. 作为开发者，我想分支名预填当前生效的默认值（`init.defaultBranch`，未配置则 `main`），这样默认情况一次确定。
8. 作为开发者，我想初始化时可以填远程仓库地址，这样 origin 一步配好。
9. 作为开发者，我想填了远程地址后自动 fetch 一次，这样远程已有的历史立刻出现在图谱里。
10. 作为开发者，我想 fetch 失败（认证/网络）时仓库照常建好、只提示 fetch 的错误，这样初始化不被网络问题连坐。
11. 作为开发者，我想两个字段都留空时行为等同裸 `git init`，这样不被强迫做选择。
12. 作为开发者，我想初始化成功后 Git Tab 自动进入空仓库态，这样无需手动刷新。
13. 作为开发者，我想空仓库有改动时看到未提交更改行并选中它进提交面板，这样 0 个提交也能暂存和提交。
14. 作为开发者，我想空仓库无改动时看到「此仓库还没有任何提交」的提示，这样知道当前无事可做。
15. 作为开发者，我想首次提交成功后图谱长出根提交、一切恢复常态，这样空仓库只是个过渡。
16. 作为开发者，我想空仓库下「修正上次提交」被禁用并提示原因，这样不会误触发注定失败的 --amend。
17. 作为开发者，我想空仓库下仍能勾选「推送」，提交成功让分支出生后再弹推送对话框，这样首次提交 + 推送一气呵成。
18. 作为开发者，我想 detached HEAD 下推送依旧禁用且提示准确，这样两种「没有当前分支」不混淆。
19. 作为开发者，我想点项目树的 "+" 时在「新建项目…」和「添加现有项目…」之间选择，这样从零开项目不用先去 Finder。
20. 作为开发者，我想「新建项目…」在一个系统面板里完成选位置和起名，这样新文件夹创建即登记。
21. 作为开发者，我想拖入文件夹的添加方式保持不变，这样已有习惯不受影响。

## Implementation Decisions

- **仓库状态检测（三管齐下）**：① 非仓库项目持有一个探测 watcher，只关心项目根下 `.git` 的出现（文件监听库无法监听尚不存在的路径，故非递归监听项目根、回调里只放行 `.git` 相关事件）；② Git Tab 每次变为可见时重探测一次；③ 非仓库兜底态提供手动「刷新」。三者与既有的仓库 watcher 构成三态对齐：非仓库（探测 watcher）↔ 仓库（.git 白名单 + 工作区双 watcher），由既有的「期望集合对齐」模型统一管理。
- **仓库根缓存失效**：缓存从「永不失效」改为「每次仓库变化重验」——watcher 防抖回调触发时（及 ②③ 入口）重新执行真实的仓库根解析并与缓存比对；不变则零额外行为，变了则更新缓存 → watcher 三态对齐 → 推仓库变化事件，渲染端软刷新后自然落到正确的态。`.git` 删除的反向场景由同一机制覆盖，落回非仓库态而非错误态。
- **初始化入口**：非仓库兜底态 = 提示文字 +「初始化 Git 仓库」主按钮 +「刷新」次按钮（VS Code 形态）；不做第二入口（右键/工具栏）。
- **初始化对话框**：进入既有 git 对话框体系，确认按钮「确定」，文案从简。两个可留空字段：初始分支名（打开时预填 `init.defaultBranch`，未配置预填 `main`；字段非空一律 `git init -b <值>`——所见即所得，需 git ≥ 2.28；清空则裸 `git init`）；远程仓库地址（非空则 init 成功后 `git remote add origin <地址>` 再 `git fetch origin`，远程名固定 origin，不提供改名输入——改名走既有仓库设置）。
- **初始化动作通道**：git 写操作判别联合新增 init 变体，是唯一合法作用于非仓库的动作——在动作层的仓库根判空之前特判，cwd 用项目路径本身。复合序列 init → remote add 中途失败即停；fetch 失败不算 init 失败（部分成功也推刷新的既有机制兜底，仓库照常进入空仓库态、单独报 fetch 错误）。
- **空仓库图谱（HEAD 未出生路径）**：加载第一步显式探测 HEAD 是否出生（rev-parse 校验，与分支/远程/贮藏并行），未出生走专用路径——log 不带 HEAD，fetch 到的远程提交照常入图（顺带覆盖 orphan 分支检出），空仓库态 = 确实一条提交都没有。不沿用原实现「匹配 log 错误文案 + 分支列表为空」的判定：前者在 log 容错损坏引用后不再成立，后者会因 fetch 出的远程分支而误入错误态。有改动就合成未提交更改行（无 HEAD 时相对空树、不锚定任何提交，git 的 status/diff --cached/add/commit 在空仓库天然可用），插在图谱最前；无改动且无提交维持「此仓库还没有任何提交」提示。未提交更改行「仅有改动时才出现」的语义不变。
- **仓库损坏容错与防造**：fetch 中断（如传输期间退出应用杀掉子进程）可能留下「引用已写、对象缺失」的半损坏仓库，此前一条坏引用即让整个图谱与提交面板不可用。三层处理：① log 恒带忽略缺失开关，损坏引用静默跳过；② 引用清单命令 fatal 时降级为单取 HEAD，图谱与未提交行照常、仅缺分支/标签标注；③ 所有 fetch（含初始化串联的）在 git 支持时加原子开关，引用更新事务化，中断不再产出坏引用（不支持的旧版 git 静默退回普通 fetch）。
- **HEAD 未出生时的动作面**：未提交行右键菜单的「贮藏」「重置」以 HEAD 为基准、git 无法执行，无 HEAD 时不显示（仅留「清理未跟踪文件」）。
- **提交面板空仓库适配**：「修正上次提交」在无 HEAD 时禁用（提示「还没有提交可修正」）；「推送」的可用判定从「有当前分支」改为「有当前分支或空仓库」（detached HEAD = 无当前分支但有 HEAD，空仓库 = 无 HEAD，两者由此区分、提示文案各自准确）；提交后弹推送对话框改为提交成功、软刷新落地后从最新状态取当前分支——首次提交后取到的就是刚出生的分支。
- **空仓库工具栏**：一个字不改（刷新/设置/视图选项本就有用；其余按钮点了失败走既有错误报告，空仓库是短暂过渡态，不做逐按钮适配）。
- **添加项目双入口**："+" 从直接弹文件夹选择器改为下拉菜单（复用既有菜单组件）：「新建项目…」= 系统 save 面板（标题「新建项目」、按钮「创建」）选位置输名字 → 主进程创建目录 → 走既有登记流程；「添加现有项目…」= 既有系统 open 面板流程。拖入不变；空态文案同步微调。新建项目不自动 git init（Git Tab 一键初始化已够近，保持正交）。

## Testing Decisions

沿用「只测纯函数、零 mock」约定，先例为主进程各模块与 shared 的既有测试：

- init 动作的参数构造纯函数（分支名空/非空、远程地址空/非空的命令序列组合）；
- 空仓库加载分支的结果组装（有改动合成未提交行、无改动维持空仓库态）；
- 推送可用性判定（当前分支/空仓库/detached HEAD 三态）作为纯函数测试；
- watcher 三态对齐逻辑中可提纯的部分（期望集合计算）。

不写 UI/e2e 测试；watcher 的真实文件系统行为靠手动验证。

## Out of Scope

- 克隆远程仓库作为第三个添加入口（值得独立特性，本次不搭车）
- 新建项目时自动 git init
- 空仓库态工具栏的逐按钮禁用适配
- 初始化对话框里的远程名自定义（固定 origin）
- unborn branch 的特殊推送逻辑（首次提交后走常规推送）
- git < 2.28 下 `-b` 参数的兼容降级（报错即报错，用户可清空分支名字段绕过）
