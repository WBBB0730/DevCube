# 多项目运行器（复刻 WebStorm Run 部件）

## Problem Statement

我经常同时在多个本地项目之间来回切换，不断地启动/停止它们的脚本——dev server、build watch、一些一次性命令。WebStorm 的运行功能是"一窗口一项目"的，要同时照看好几个项目就得开一堆窗口、开一堆重型 IDE。我没有一个轻量的地方能**把所有项目和它们能跑的命令放在一起看**、一次拉起好几个、并盯着各自的实时输出。

## Solution

一个轻量桌面工具，在**单个面板**里聚合多个 **Project**。对每个 Project 自动把它 `package.json` 里的脚本呈现为随时可跑的 **Discovered Script**，也允许我保存自定义命令。点一下就用**真正的终端控制台**（彩色、spinner、可输入）跑起来；多个 **Run Session** 可并发，每条配置显示自己的状态；选中谁就在右侧看谁的实时输出。整体复刻 WebStorm 右上角 Run 部件的手感（运行 / 重新运行 / 停止），但把它从单项目扩展到**多项目并列**。

## User Stories

1. 作为用户，我想把一个本地文件夹添加为 **Project**，以便在面板里管理和运行它的命令。
2. 作为用户，我想通过"添加项目"按钮打开系统文件夹选择器来添加 Project，以便快速纳入一个项目。
3. 作为用户，我想添加成功或再次添加**已有项目**后都**自动选中该项目**并在必要时滚入视口，以便立刻在右侧看到它、不必再在列表里找。
4. 作为用户，我想直接把文件夹**拖进面板**来添加 Project，以便省去点选。
5. 作为用户，我想添加**任意文件夹**（哪怕没有 `package.json`）为 Project，以便 Python/Rust/Go 等非 Node 项目也能靠命令型配置运行。
6. 作为用户，我想在面板左侧**同时看到多个 Project 并列**，以便一眼掌握我在照看的所有项目。
7. 作为用户，我想通过"更多"菜单**移除 Project**，以便清理不再需要的项目。
8. 作为用户，我想通过项目行的「更多」菜单或右键菜单**打开文件夹**，以便在系统文件管理器中查看该项目目录。
9. 作为用户，我想通过项目菜单的「打开于」在 **Claude / Codex / Cursor** 桌面端以该 Project 根目录开新会话，以便无缝跳到外部编码工具；未安装的项应置灰并说明原因，而不是静默消失。
10. 作为用户，当 Project 有顶层 `package.json` 时，我想自动看到它的 scripts 作为 **Discovered Script** 候补，以便零配置直接开跑。
11. 作为用户，我想让 Discovered Script **随 `package.json` 实时增删**，以便它始终反映当前的 scripts。
12. 作为用户，我想让一条 Discovered Script 在**第一次运行后自动"晋升"**为一条 Run Configuration，以便我常用的那几条自动沉淀到"我的配置"。
13. 作为用户，我想让晋升后的（引用型）配置**从候补区消失**、只在"我的配置"里出现一次，以便同一条 `dev` 不重复展示。
14. 作为用户，我想让引用型配置**始终引用 `package.json` 里的那条 script**、随其变化同步，以便我改了 script 后它下次自动跟着变。
15. 作为用户，我想让引用型配置**完全不可自定义**（名字即 script 名，无自定义命令/cwd/env），以便它保持"就是这条 script"的纯粹语义。
16. 作为用户，当引用型配置引用的 script 从 `package.json` 消失时，我想让它**自动删除**，以便"我的配置"里不留悬空项，且不会丢失任何我手工填的内容。
17. 作为用户，我想**新建一条命令型 Run Configuration**（命令行 + 工作目录 + 环境变量），以便运行任意命令（如 `docker compose up`）。
18. 作为用户，我想给命令型配置设置**相对项目根的自定义 cwd**，以便在子目录里运行命令（也用作 monorepo 子包的轻量绕过）。
19. 作为用户，我想给命令型配置设置**环境变量**，叠加在继承的环境之上，以便注入 `NODE_ENV` 之类。
20. 作为用户，当我想给某条 script 加环境变量/改 cwd/改命令时，我想**新建命令型配置**来承载定制，以便引用型保持纯净、我的定制独立持久化且永不被自动删。
21. 作为用户，我想给每条配置一个**运行按钮**和一个**更多按钮**，以便快速启动或访问其操作。
22. 作为用户，我想右键配置行打开与「更多」相同的菜单（编辑仅命令型 / 删除；运行中亦可），以便不必找⋮按钮。
23. 作为用户，当一条配置正在运行时，我想让运行按钮变为**重新运行**、更多按钮变为**停止**，以便与 WebStorm 行为一致。
24. 作为用户，我想让**重新运行**先停掉旧进程再起新的（单实例），以便 dev server 这种"我只想要一个"的场景不会起重。
25. 作为用户，我想让**不同配置的 Run Session 并发运行**，以便同时开着多个项目的 dev server。
26. 作为用户，我想在右侧的**单个控制台**里看到左树当前选中项的会话输出，以便聚焦查看某一个的日志。
27. 作为用户，我想让控制台**跟随左树选择**切换，以便点谁看谁、无需管理一堆 tab。
28. 作为用户，我想让会话**进程退出后仍保留控制台内容**并显示退出状态，以便回看跑挂了的最后日志。
29. 作为用户，我想让未被选中的后台会话**继续缓冲输出**，以便切回去时能看到完整历史。
30. 作为用户，当右侧激活的是某条配置的**运行会话 Tab**时，我想在 Tab 栏下方看到一行**操作栏**（运行/重跑、停止、清空输出），以便不必回到左树就能控制该会话并清掉刷屏日志。
31. 作为用户，我想在控制台里**向运行中的进程输入**（PTY），以便回答 `y/n` 提示或与 REPL 交互。
32. 作为用户，我想让控制台**正确渲染 ANSI 颜色、spinner、进度条、`\r` 覆盖行**，以便 vite/webpack 的输出原样好看。
33. 作为用户，我想让 Discovered Script 用项目**探测到的包管理器**运行（由 lockfile 决定 pnpm/yarn/bun/npm），以便命令与我在终端里跑的一致。
34. 作为用户，我想让命令通过**登录 shell** 启动，以便 `nvm`/自定义 PATH 被正确加载，不再 `command not found`。
35. 作为用户，我想让**停止**杀掉整棵进程树，以便 dev server fork 出的子进程不残留为孤儿。
36. 作为用户，我想在每条配置上看到**状态点**（空闲 / 运行中 / 成功退出 / 失败退出），以便一眼判断哪个在跑、哪个挂了。
37. 作为用户，我想让工具**监听每个项目的 `package.json`（及 lockfile）**并即时更新候补与包管理器，以便我改了 scripts 立刻生效。
38. 作为用户，我想在**重开 App 时恢复项目列表和我的配置**，以便接着上次继续。
39. 作为用户，我**不希望**重开 App 时自动重跑上次在跑的会话，以便打开时不会一下子起一堆 server。
40. 作为用户，我想让配置**集中存储、不写进任何项目仓库**，以便不污染我的 repo、也不和 WebStorm 的 `.idea/` 冲突。

## Implementation Decisions

**进程/渲染分工。** 主进程持有全部有状态、有权限的部分：Project 注册表与配置存储（集中存 `userData` 的一份 JSON，见 ADR-0002）、`package.json`/lockfile 的文件监听、每个 Run Session 的 PTY 进程（见 ADR-0001）、由 lockfile 探测包管理器、以登录 shell 启动命令、以及"杀整棵进程树"的停止。渲染进程只负责 UI：左侧项目树、右侧以项目为维度的 Tab 栏（Git / 运行会话 / 终端）与 xterm、**激活运行会话 Tab 时 Tab 栏下方的操作栏**（运行·停止·清空），以及命令型配置的新建/编辑表单。清空输出经主进程同步清无头终端缓冲并换代 sid，避免切回时回填旧内容。

**数据模型。**

- **Project**：以**绝对路径**为标识的文件夹。
- **Discovered Script**：从该 Project 顶层 `package.json` 的 `scripts` 实时派生，**不持久化**；只看顶层，不展开 workspace。
- **Run Configuration**，两型：
  - **引用型**：只存 `(Project, script 名)` 的引用；运行时才从 `package.json` 解析命令、随其同步；无任何可自定义字段；引用的 script 消失即删。
  - **命令型**：存命令行 + 可选 cwd（相对项目根解析，默认根）+ 环境变量（叠加于继承环境）；完全独立，不引用、不随 script 变，永不自动删。
- **Run Session**：仅存在于主进程的临时执行实例；一条配置**至多一个活跃会话**（单实例）；不同配置可并发。

**关键规则。**

- **晋升**：运行一条 Discovered Script 即把它转成一条引用型 Run Configuration，按 `(Project, script 名)` 去重，候补区不再展示它。
- **对账（reconcile）**：`package.json` 变化时和每次 App 启动时都重新派生 Discovered Script 并比对——新增的进候补，消失的对应引用型配置直接删除，script 改名视作"删旧出新"。
- **执行**：引用型跑 `<探测到的 PM> run <script 名>`，cwd 恒为项目根（避免"解析哪个 package.json"的歧义）；命令型跑用户命令、cwd 按相对项目根解析；两者都经登录 shell 起 PTY。
- **重新运行**：先杀整棵进程树再重启，复用同一控制台。

**IPC 契约（方向性，非具体签名）。** 渲染→主：添加/移除 Project、列出某 Project 的 Discovered Script 与 Run Configuration、运行/停止/重新运行某配置、向会话写 stdin、通知 PTY 尺寸变化、新建/编辑/删除命令型配置、**打开项目文件夹**（既有 `shell.openPath`，仅放行已登记项目路径）、**打开于外部编码桌面工具**（探测安装状态；仅放行已登记项目根；Claude / Codex / Cursor，见 ADR-0018）。主→渲染：会话输出增量、会话状态变更（运行中/成功/失败）、Discovered Script 增删、引用型配置被自动删除。

**打开于。** 项目「更多」/右键菜单提供「打开于」子菜单（字母序：Claude、Codex、Cursor）。打开菜单时探测本机是否安装对应桌面端（应用路径或官方 CLI）；未装项置灰并 hover 说明原因。唤起方式见 ADR-0018：各工具走官方推荐入口，跨 macOS / Windows / Linux 适配。子项前缀品牌图标：Claude 用 Claude Desktop App 图标、Codex 用官方浅色标（蓝云 `>_`）、Cursor 用 Dock 同源 App icns（含 squircle 遮罩）。

**文件监听。** 只监听每个 Project 的顶层 `package.json` 与 lockfile（不监听整个项目树，避免 `node_modules` 事件风暴），采用能稳妥处理编辑器原子写（写临时文件 + rename）的跨平台监听方案；具体库在落地时按项目 ctx7 规矩确认当前 API 后选用。

## Testing Decisions

**好测试的标准。** 只测外部可观察行为，不测实现细节。绝大多数值得测的逻辑是主进程里的**纯领域规则**，应把"起进程"这件事抽象为可替换的 spawner，从而无需真的拉起进程即可测。

**要测的模块/行为：**

- **晋升与去重**：运行 Discovered Script 后它变成引用型配置、并从候补区按 `(Project, script 名)` 去重消失。
- **对账**：`package.json` 增/删/改名 script 时，候补新增、引用型配置在 script 消失时被删、改名走"删旧出新"；App 启动时的一次性对账等价于监听路径。
- **包管理器探测**：不同 lockfile 组合下解析出正确的 PM，无 lockfile 时回退 npm。
- **引用型命令解析与同步**：引用型配置在运行时按当前 `package.json` 解析出正确命令，script 改了内容后随之改变。
- **命令型 cwd 解析**：相对项目根正确拼接，默认根。
- **单实例语义**：对运行中的配置再次运行等价于"停旧起新"，且同一配置不产生两个活跃会话；不同配置可并发。
- **打开于**：deep link / CLI 参数拼装与「有应用路径或 CLI 即视为可打开」的探测规则（注入假 fs / PATH，不真启外部 App）。

**会话生命周期与控制台/PTY** 属于集成层：用假 spawner 驱动会话状态机（运行中→成功/失败、停止→杀树）来测状态流转；真实 PTY 渲染与真实进程拉起不作为单测重点。

**先例。** 本仓库为全新脚手架，暂无同类测试；本特性将确立"主进程领域逻辑 + 可替换 spawner"的测试范式，供后续沿用。

## Out of Scope

- 调试器（断点、attach）——只做 Run，不做 Debug。
- Before-launch 任务（用命令型里的 `a && b` 顶替）。
- `.env` 文件加载（只支持每条命令型配置内联填 env）。
- OS 系统通知（靠左树状态点提示崩溃/退出）。
- **Compound / 一键启动多个**（最贴合聚合诉求，列为紧随其后的下一迭代，但不在本 MVP）。
- 控制台内搜索、项目分组。
- monorepo / workspace 自动展开（子包各自作为 Project 添加，或用命令型配置的自定义 cwd 绕过）。
- 配置的仓库内共享/提交、跨机同步。

## Further Notes

- **绝对路径标识**意味着移动项目文件夹会使其配置失联；"重新定位"能力是已知的后续增强点。
- **Compound** 是最可能的第二迭代，因为它直接放大"多项目聚合"的核心价值；届时需引入"会话组 / 部分失败"语义，会扩展当前模型。
- 文件监听方案可随时替换，不构成架构锁定。
- **视觉方向**（DESIGN.md）尚未定义：本次只锁定了交互方向（左树 + 右单控制台、跟随选择、运行/更多两按钮、状态点、退出后保留输出），配色/密度/字体等留待后续。
