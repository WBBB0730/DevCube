## Problem Statement

我在 DevCube 里聚合了多个项目来运行命令、看输出，但想了解某个项目的 git 状态（历史走向、分支、某次提交改了什么、要不要合并/回滚）时，必须切去 IDE 或终端。项目维度的工作台缺一块「仓库视图」，上下文来回切换很割裂。

## Solution

每个项目的 Tab 栏常驻一个 **Git Tab**（排最前、不可关闭）：完整移植 vscode-git-graph 的能力——提交图谱（分支线 + 圆点 + 引用标签）、提交/贮藏/未提交更改的详情与文件变更、应用内 diff 面板、右键执行全套 git 操作（分支/标签/合并/变基/重置/拣选/贮藏/远程同步）、分支筛选、提交查找、仓库设置。数据由主进程直接跑 git CLI 产出；`.git` 与工作区文件变动自动刷新图谱。

## User Stories

1. 作为开发者，我想在项目的 Tab 栏里直接打开 Git Tab，这样不用离开 DevCube 就能看仓库历史。
2. 作为开发者，我想看到提交历史的图形化分支线与每行的分支/标签/贮藏标签，这样一眼读懂仓库拓扑。
3. 作为开发者，我想看到「未提交的更改 (N)」行，这样知道工作区是否干净。
4. 作为开发者，我想点一个提交展开详情（作者/提交者/日期/完整信息/改动文件树），这样了解这次提交做了什么。
5. 作为开发者，我想点详情里的文件直接看 diff（应用内渲染），这样不依赖外部编辑器。
6. 作为开发者，我想 Ctrl/Cmd+点第二个提交进入比较模式，这样看两点之间的全部文件差异。
7. 作为开发者，我想右键提交创建分支/打标签/拣选/回滚/丢弃/重置，这样常用操作不用敲命令。
8. 作为开发者，我想右键分支标签执行检出/重命名/删除/合并/变基/推送，这样分支管理都在图上完成。
9. 作为开发者，我想双击本地分支标签直接检出、双击远程分支标签创建跟踪分支，这样高频操作最短路径。
10. 作为开发者，我想右键远程分支执行拉取/获取到本地分支/删除远程分支。
11. 作为开发者，我想对贮藏执行应用/弹出/丢弃/建分支，并在图上看到贮藏节点。
12. 作为开发者，我想右键未提交行进行贮藏/重置/清理未跟踪文件。
13. 作为开发者，我想用工具栏的分支下拉筛选图谱显示的分支（多选、显示全部）。
14. 作为开发者，我想勾选「显示远程分支」开关控制远程引用的显示。
15. 作为开发者，我想点「刷新」从所有远程 fetch 并重新加载图谱（本地先出、远程随后）。
16. 作为开发者，我想 Cmd/Ctrl+F 搜索提交（消息/hash/作者/引用，支持大小写与正则），命中行高亮并可跳转。
17. 作为开发者，我想在仓库设置里管理 remote（增删改/清理）、编辑 user.name/email、隐藏某些 remote。
18. 作为开发者，我想让危险操作（删除分支/重置/丢弃提交等）先弹确认对话框，避免误操作。
19. 作为开发者，我想在分支未合并导致删除失败时得到「强制删除」的追问，而不是原始报错。
20. 作为开发者，我想在推送标签时得到「提交不在远程」的预警。
21. 作为开发者，我想操作失败时看到 git 的原始错误信息。
22. 作为开发者，我想 `.git` 目录或工作区文件变化（外部提交/切分支/编辑文件）后图谱与未提交明细自动刷新，且刷新不打断我展开的详情。
23. 作为开发者，我想图谱「加载更多」或滚动到底自动加载更早的提交。
24. 作为开发者，我想对非 git 仓库的项目看到明确的兜底提示而不是报错。
25. 作为开发者，我想交互式变基在项目的 Terminal Tab 里启动，这样可以在终端里完成编辑。
26. 作为开发者，我想注释标签能查看 tagger 与消息详情。
27. 作为开发者，我想从提交详情里打开工作区文件、复制文件路径。
28. 作为开发者，我想把某文件重置到某个提交时的状态。
29. 作为开发者，我想在工具栏左侧的「视图选项」里切换图谱显示（远程分支/贮藏/标签/reflog/首父/提交排序），这样不必进仓库设置弹窗。

31. 作为开发者，我想从工具栏直接刷新（fetch + 静默重载，本地先出、远程随后）、拉取、推送（可强制）、创建分支，这样常用远程/分支操作不必进右键菜单。

34. 作为开发者，我想暂存某文件后继续修改它时，已暂存区仍是暂存那一刻的快照、未暂存区显示后续改动，这样部分暂存语义正确。
35. 作为开发者，我想点已暂存文件看 HEAD→暂存 的差异、点未暂存文件看 暂存→工作区 的差异。
36. 作为开发者，我想写提交信息后「提交」，或勾「修正」改上次提交，或用「提交并推送」一步同步（推送时可选强制），这样提交与推送连贯完成。
37. 作为开发者，我想从文件行的 `…` 菜单撤销更改 / 删除未跟踪文件 / 打开文件（进入该项目 **Files Tab**）/ 复制路径。
38. 作为开发者，我想暂存 / 取消暂存 / 撤销都静默即时、不打断我看图，这样操作顺滑。
39. 作为开发者，我想在拉取对话框里选择 remote 与远程分支（分支旁有「刷新」现场 fetch），这样没有上游的分支也能从工具栏拉取。
40. 作为开发者，我想在拉取时选择整合方式（合并 / 变基 / 压缩提交），且对话框记住我上次的选择（首次跟随 git 配置），这样习惯变基的我不用每次都改。
41. 作为开发者，我想在推送对话框里选择本地分支和目标远程分支（可输入尚不存在的新分支名），这样能完成「dev 推到 origin/main」这类操作。
42. 作为开发者，我想推送时勾选「同时推送该分支上的标签」（附注与轻量都推），这样发版打的标签不必逐个推送。
43. 作为开发者，我想多 remote 仓库的「刷新」不再必然失败（--atomic 与 --all 互斥），且中断安全性不打折。
44. 作为开发者，拉取/合并/变基/拣选/回滚撞上冲突后，我想在 Git Tab 里看到一条明确的「操作进行中」状态条，而不是只有一次性的错误框。
45. 作为开发者，我想在状态条上直接 继续/跳过/中止 进行中的变基（拣选/回滚同理），合并则可中止或去提交面板完成提交，不必开终端。
46. 作为开发者，操作进行中时我想让会撞车的动作入口置灰并注明原因，避免再撞出「已存在 rebase-merge 目录」这类 fatal。
47. 作为开发者，我想在提交面板里一眼认出冲突文件（红「冲突」标），勾选即把它标记为已解决。
48. 作为开发者，我想点冲突文件时能看到其带冲突标记的差异内容（或明确的兜底说明），而不是文件凭空消失。
49. 作为开发者，我想在没有任何改动时也能从工具栏打开提交面板（例如只勾「修正上次提交」改提交信息），且提交完全部改动后面板不被强制关闭。
50. 作为开发者，我想在 Git Tab 的标签上始终看到当前分支名（用括号包住，如 `Git (main)`；detached HEAD 时显示缩写 hash），不必先点开 Git Tab，这样不用切到 Git Tab 也知道项目在哪个分支。

## Implementation Decisions

- **入口与 Tab 模型**：Git Tab 是每项目常驻、非会话的 Tab，键 `git:<projectPath>`，恒排最前、不可关闭；与会话 Tab 共用激活/循环/回落规则（ADR-0005）。面板「切走隐藏不卸载」。项目成为当前项目时即全量预加载（仅 idle 触发），Tab 标签始终能显示当前分支名（括号包住，如 `Git (main)`；detached HEAD 回落缩写 hash）；打开 Git Tab 时若已就绪则只重验仓库根。此后随 git:changed 软刷新保鲜。
- **数据源**：主进程直接 spawn git CLI（不经 shell），git 路径经用户登录 shell 解析一次并缓存（GUI 应用 PATH 缺失）；所有解析基于机器可读输出（自定义 --format 分隔符、-z NUL 分隔）；解析器是纯函数、与 IO 编排分层。
- **对齐策略**：UI 层为 React 重写（原项目是命令式 DOM + webview，不可复用）；**纯逻辑层以参考项目原源码为唯一权威**——git 命令构造、解析规则、布局算法、菜单可见性、对话框默认值逐函数对照原实现，能照抄的函数体照抄（仅剥 vscode 依赖、调整类型），行为偏差按缺陷修正。
- **右键菜单**：与项目树共用 `ui/context-menu`（Base UI ContextMenu + Backdrop）。图谱行 / 引用标签 / 详情与提交面板文件行等多入口经 store 记录目标与坐标后受控打开，Positioner 用虚拟 anchor（无法用单一行级 Trigger 包住全部入口）；菜单项可见性仍由纯函数 `buildMenuItems` 产出。
- **IPC 契约**：读操作独立通道（load/details/file-diff/tag-details/repo-config/settings）；约 30 个写操作走单通道判别联合 GitAction，结果为判别联合（ok / error / push-tag-not-on-remote）；仓库变化推 git:changed 事件，渲染端只对已加载过的项目软刷新。
- **加载语义**：一次 load 主进程内部串联仓库概要 + 提交列表 + refs 返回；初始 300 条、加载更多 +100 后整表重拉（哨兵 N+1 判断还有更多）；软刷新保持画面原地换数据，硬刷新（筛选/设置变更/重试）清空重来；空仓库与非仓库是明确的状态而非错误。
- **图谱布局**：忠实移植参考项目的 Vertex/Branch/determinePath 算法为纯函数（网格坐标 → SVG 路径），12 色调色板循环，颜色经 CSS 变量引用。
- **详情与 diff**：详情面板 v1 采用吊底停靠（非参考默认的行内嵌入——React 表格行内嵌套复杂度高，作为已知取舍）；diff 面板由 `@git-diff-view/react` 渲染（语法高亮、词级 diff、统一/左右对比内置；无虚拟滚动，大文件全量渲染，见 ADR-0007），数据为主进程透传的原始 unified diff 文本（git diff -U3，二进制判定仍在主进程）；比较方向恒按行序归一化（老→新），未提交端恒为 to。文件行一律可点开 diff（未跟踪目录整体条目除外）——是否二进制由点开后的 diff 正文判定并在面板内说明，列表阶段不预判；未跟踪文件的行数由主进程读工作区文件补算（numstat 口径，二进制/超大文件保持无统计）。二进制中的图片文件（png/jpg/gif/webp 等，svg 走文本 diff）在 diff 面板内给新旧预览（object-contain 带边距，M 双栏对照、A/U/D 单栏；数据经 gitFileImage 通道按需取，端点语义同 gitFileDiff，旧端在「提交自身变更」场景取 hash^）；无 hunk 的 diff（纯重命名 / 模式变更等）统一显示「没有差异内容」。工作区监听（ADR-0021，@parcel/watcher 与 Files/discovery 共用订阅）忽略规则零硬编码：`.git` 仅放行白名单元数据（HEAD/index/config/refs），工作区事件路径在防抖到期后经 `git check-ignore` 批量过滤（.gitignore / info/exclude / 全局 excludesfile 全部由 git 裁决），全部被忽略的事件不触发刷新。
- **写操作安全**：动作对话框与追问链对齐参考（强删确认、重名替换、push tag 预检）；「不再提示」标记持久化于跨项目视图偏好；动作执行期间项目文件监听**全部通道**静音（discovery / files / git，含 1500ms 余震），完成后主动推刷新。
- **凭证**：依赖系统凭证助手（ssh-agent/钥匙串），注入 GIT_TERMINAL_PROMPT=0 快速失败，错误原样呈现；不做 askpass。
- **交互式变基**：不在主进程跑——渲染端在该项目新建 Terminal 并写入 `git rebase --interactive` 命令行。
- **持久化**：每项目 git 设置（显示开关三态/排序/隐藏 remote）与跨项目视图偏好统一存 electron-store（ADR-0002 同一份 JSON）。
- **不做头像**（无网络依赖）；**不做 GPG 签名校验**（tag 详情仅展示是否签名）。
- **工具栏重排**：右侧操作钮 = 刷新 / 拉取 / 推送 / 创建分支 / 搜索 / 设置。「刷新」= fetch 全部远程 + 静默软重载（本地图谱先原地更新、远程 fetch 完成后再软刷新一次；全程不置全局 loading、不弹进行中遮罩），取代原独立「获取」钮；git ≥ 2.31 时逐个 remote 执行 `git fetch <remote> --atomic`（`--atomic` 与 `--all` 互斥，逐个执行在多 remote 下保住「中断不留损坏引用」；不理会罕见的 `remote.<name>.skipFetchAll`），旧版 git 维持 `fetch --all`。拉取 / 推送 / 创建分支复用既有对话框，预设为当前分支 / HEAD；推送在 detached 或无 remote 时置灰，拉取同此条件（无上游不再置灰——目标在对话框内选）；推送对话框已含 常规 / force-with-lease / 强制。
- **拉取 / 推送对话框表单化**：两对话框从「确认式」升级为「表单式」。拉取 = remote 单选（默认上游 remote，其次首个）+ 远程分支单选（默认上游分支，其次同名，再次留空；选项来自已加载 refs）+ 分支旁「刷新」钮（fetch 所选 remote，期间禁用对话框按钮）+「整合方式」单选：合并（附「即使可以快进也创建新的合并提交」勾选，仅此项生效）/ 变基（提示会重写本地未推送的提交）/ 压缩提交；整合方式默认值 = 记住上次选择（跨项目视图偏好）> git 配置（`branch.<name>.rebase`，其次 `pull.rebase`）> 合并。推送 = remote 由多选改单选（默认逻辑沿用）+ 本地分支单选（默认按入口预设）+ 目标远程分支可输可选（默认与本地分支同名并随其切换自动跟随，手动编辑过即不再跟随；旁置「刷新」钮同拉取）+「同时推送该分支上的标签」勾选（默认不勾；`git tag --merged <本地分支>` 计算该分支历史上的全部标签——附注与轻量都含——以 refspec 并入同一条 push）+ 推送模式三档不变；目标名 ≠ 本地名时「设置上游」默认不勾（避免偶发的 dev→main 推送改掉 dev 的上游），同名时默认勾。
- **视图选项 Popover**：图谱显示三态开关（远程分支/贮藏/标签/reflog/首父）+ 提交排序从「仓库设置」移出，收进工具栏**左侧、紧挨分支下拉**的 Popover（工具栏原重复的「显示远程分支」checkbox 并入）。分支下拉浮层项「显示全部」改文案「全部分支」，并加「当前分支」= filter 传 `HEAD` ref 动态跟随（不持久化，随重启回落全部）。
- **仓库设置瘦身**：仅保留 git 级配置（用户信息、远程管理、隐藏远程）。删除死配置「显示名」（从不被读）与「Issue 链接」整功能（含 `tokenizeBody` 的 issue 层、`IssueLinkingConfig` 类型/字段/校验）；提交信息中的 URL / 邮箱自动链接保留。
- **日期中文化**：`formatDateTime` 改用 `Intl.DateTimeFormat('zh-CN')` 输出「YYYY年M月D日 HH:MM:SS」（零依赖标准 API，去掉手写月份表）；提交表格「日期」列仍为中文相对时间。
- **操作进行中（冲突）处理**：load 时检测 gitdir 下 rebase-merge / rebase-apply / MERGE_HEAD / CHERRY_PICK_HEAD / REVERT_HEAD（经 `git rev-parse --git-path` 定位，worktree 亦稳），得 `opInProgress: rebase | merge | cherry-pick | revert | null` 随加载结果下发；状态文件变化经既有 `.git` 监听自动跟进。工具栏下方出常驻状态条：变基/拣选/回滚 = 文案 +「继续 / 跳过 / 中止」，合并 = 文案 +「中止」+ 提示去提交面板完成提交；继续/跳过为新 GitAction（命令注入 GIT_EDITOR=true 防编辑器挂起），中止走既有危险确认框；cherry-pick / revert 的 --skip 版本 gate git ≥ 2.23。防误触：opInProgress 非空时拉取 / 合并 / 变基 / 拣选 / 回滚 / 检出 / 贮藏应用与弹出等会撞车的入口置灰（hover 说明原因），推送与只读操作不受影响。冲突文件：porcelain XY 任一位为 U 或 AA/DD 判冲突（现有 diff --cached / diff 走 --diff-filter=AMDR，冲突文件天然缺席，须从 status 单独成桶），进未提交明细独立桶；提交面板未暂存段以红「冲突」徽标标出，勾选 = `git add` = 标记已解决；冲突文件 diff 尽力渲染，combined diff 库吃不下时按「二进制/无差异」同款兜底文案。
- **未提交更改的暂存与提交**（见 ADR-0006）：未提交行详情从只读升级为带暂存区的提交面板（版式 A：左提交信息框 + 「提交」/下拉「提交并推送」+「修正」勾选，右已暂存/未暂存两段文件树）。数据模型改为 `git diff --cached`（已暂存 HEAD↔index）+ `git diff`（未暂存 index↔工作区）+ 未跟踪，同一文件可同时在两段；diff 面板支持 index 端点。勾选 = `git add`、取消勾选 = unstage、区头 = 全部；写操作 stage/unstage/discard/commit(amend) 静默即时，提交提交 index，「提交并推送」提交后打开 push-branch 对话框（含强制）。仅文件级暂存。文件行 `…` 菜单：未暂存 = 撤销更改…（未跟踪则删除文件…）/ 打开文件（进入该项目 **Files Tab**，见 `files-tab` PRD；「在其他应用中打开」改由 Files Tab 提供）/ 复制路径；已暂存 = 取消暂存 / 打开文件 / 复制路径。图上未提交行的行级右键菜单（贮藏/重置/清理）不变。提交面板不随「未提交更改」行消失而收敛关闭（目标是工作区、恒不过期）：工具栏提交钮在无改动时也可打开面板（仅空图禁用——空仓库且无改动时详情面板无处停靠），提交/丢弃清空全部改动后面板保持打开、两段显示空占位。

## Testing Decisions

只测纯函数、零 mock（项目既有约定）：git stdout 解析器用真实样本字符串（含二进制 numstat、rename 跨 NUL 段、annotated tag 双行、空仓库等参考实现的已知坑）；动作层测参数构造纯函数与错误聚合语义；布局引擎测列分配/线段坐标/路径 d 串到具体数值；日期格式化测阈值边界。先例：src/main/*.test.ts 与 src/shared/runnable.test.ts 的「构造输入 → 断言输出」风格。不写 UI/e2e 测试。新增：暂存/未暂存/未跟踪的 git 输出解析（`git diff --cached` / `git diff` / status）用真实样本测（含同一文件同时落在两段的场景），沿用纯函数零 mock 约定；stage/unstage/commit 动作层测参数构造；日期格式化改测中文 Intl 输出的边界串。未跟踪行数补算的 countLinesInBuffer 测空内容 / 末行无换行 / NUL 判二进制。拉取/推送对话框升级新增的参数构造纯函数（pull 整合方式三态、push 的 `local:target` refspec 与标签 refspec 拼装、fetch 逐 remote atomic 展开）沿用同一风格。冲突处理新增：porcelain 冲突桶解析（UU/AA/DD/DU 等真实样本）、继续/跳过/中止的参数构造与 --skip 版本 gate 文案。监听路径分类（`project-watch-classify`）：`.git` 白名单 → git-meta；objects 噪声忽略；工作区路径 → git-worktree（是否刷新仍靠运行时 check-ignore，单测不起真实 git）。

## Out of Scope

- Pull Request 创建（提供方模板配置整块不做，设置形状留 null 占位）
- Code Review 跟踪（已读文件标记）
- 作者头像、GPG 签名校验、askpass 凭证交互
- 提交列表列宽拖拽与列显隐持久化、refs 标签三种对齐模式（v1 固定 Normal）
- 消息内 emoji shortcode / markdown 富渲染（v1 纯文本 + URL/邮箱自动链接）
- 行内嵌入式详情（v1 吊底停靠）
- 多仓库/子模块发现（每项目即其根仓库）
- Issue 链接（提交信息 issue 号链接化）—— 整功能移除（URL / 邮箱自动链接保留）
- 部分暂存（hunk / 行级 staging）—— 仅文件级
- 文件夹级复选框 —— 暂存靠单文件勾选或区头「全部」
- 分支筛选持久化（含「当前分支」模式）—— 纯内存，随重启回落「全部分支」
- 推送对话框的多分支批量勾选（一次推多个分支）与 remote 多选（改单选，要推多个 remote 分两次）
- 拉取后自动设置上游；拉取遇脏工作区自动暂存（--autostash）—— 直接呈现 git 报错
- 应用内三方合并编辑器 / 逐 hunk 冲突解决 —— 解决冲突交给编辑器/IDE，应用只负责状态呈现与继续/中止
