# 回填走主进程无头终端的屏幕快照，而非原始输出流

会话输出的回填（切换选择、渲染端刷新）曾直接重放主进程缓存的原始 pty 字节流。但原始流跨宽度重放无法保真：zsh 行尾标记（`%` + 一屏宽空格 + 回车的自擦除技巧）、SIGWINCH 重画等序列都依赖**产生时**的列宽，缓冲里只要混有不同宽度时期的片段，重放就出现残留 `%`、缺字、错位。因此主进程为每个会话维护一个 `@xterm/headless` 无头终端实时消费 pty 输出、跟随 resize，回填时用 `@xterm/addon-serialize` 下发**当前屏幕状态**（VS Code 同款方案）。

## Considered Options

- **无头终端 + serialize（选中）**：屏幕状态天然按当前宽度编码，回放完全保真；顺带取代 1MB 环形字符串缓冲（历史行数由 scrollback 界定）。代价：每会话一个无头实例的内存与二次解析 CPU。
- **原始流重放 + 按快照宽度回放**：改动小，但只对"单一宽度产生的缓冲"有效，窗口 resize / 终端诞生期的多宽度片段在原理上无法修复。

## Consequences

- 渲染端回填协议扩展（`SessionBufferSnapshot`）：`data` 语义从"原始流尾部"变为"序列化屏幕"；`bytes` 仍按原始流累计，但去重不再限于回填期间——快照回复与输出事件走不同派发通道、到达顺序无保证，快照已含的块（如运行头部，它随本方案改为与快照双路径下发）可能在回填完成后才到达，因此**所有**实时写入都必须按 `bytes` 去重。
- `bytes` 只在同一会话代内可比（重跑即换代、从零重计），事件与快照均携带代际标识 `sid`，渲染端丢弃跨代事件——否则快速重跑时旧会话的在途事件会以旧累计值污染写入进度，静默吞掉新一轮输出。
- serialize 不携带光标可见性（DECTCEM）：已结束的运行会话由渲染端在回填后补写 `\x1b[?25l`。
- 无头终端 scrollback 与渲染端 xterm 一致（10000 行），决定快照能带回的历史行数上限。
